# 浏览器知识点
## http1.0 和 http1.1

- 1.1默认使用长链接，Connection：close 关闭
- 1.1 必传 host域：主机名
- 1.1 增加了 options put delete trace connect 这些request方法
  - TRACE方法是HTTP（超文本传输）协议定义的一种协议调试方法，该方法使得服务器原样返回任何客户端请求的内容。
  - connect 用于支撑https加密
- 1.0 缺点
  - 队头阻塞 请求队列中有请求阻塞后面都等待 造成页面加载速度降低
  - 明文传输 不安全

## http2

- http2 基于 SPDY 专注于性能，最大的目标是在用户和网站间只用一个连接，该连接可以承载任意数量的双向数据流
- 二进制传输 二进制协议解析起来更高效
- header 压缩
  - 在客户端和服务器两端简历字典，用索引号表示重复的字符串 可以达到50%-90%的高压缩率
- 用户和网站间只用一个连接
  - 数据流 以消息形式发送 消息由多个1至多个帧组成 发送帧可以乱序 根据帧首部流标识可以组装
- 服务器推送 提高性能
  - 不需要客户端请求就发送资源 例如客户端只请求 index.html,服务端直接返回index.html、style.css、example.png 
- 缺点 队头阻塞 没有彻底解决 tcp丢包时还是会阻塞 因为tcp对丢包有个特别的重传机制

## http3

### QUIC

- 实现了可靠性传输 提供了数据包重传、拥塞控制以及其他一些tcp中存在的特性
- 基于udp udp无链接 不需要握手挥手 比 tcp快 大大提升页面首次打开速度
- 集成了TLS加密功能 类似于https
- 多路复用 彻底解决 对头阻塞
  - quic实现了在同一物理连接上可以有多个独立的逻辑数据流，实现了数据流单独传输，解决了tcp队头阻塞
  - 0RTT 是指建立连接之后 后续发送不需要增加额外的RTT时间，最开始握手还是需要1RTT时间消耗的

## 简单请求 和 复杂请求

### 简单请求

不涉及一些安全问题且无自定义头（例如没有token）的请求可以理解为简单请求，所以简单请求的方法是get、post、head这几种，put（修改文件）、delete（删除文件）肯定不是简单请求，简单请求会先执行后判断是否跨域，post请求头中含有content-type字段为 text/plain（纯文体传输 空格转换为“+”）、multipart/form-data（**指定传输数据为二进制类型，比如图片、mp3、文件**）、application/x-www-form-urlencoded（form表单post的默认格式）这几种。

### 复杂请求

- put、delete方法的ajax请求
- 发送json格式的ajax请求
- 带自定义头的ajax请求（例如带有token），因此post get 也会是复杂请求
- 发送预检命令 实际就是询问请求权限
  - 预检命令响应头返回access-control-max-age 作为预检命令过期标志

## 浏览器缓存

### 缓存是什么

- 是指浏览器在本地磁盘中对方问过的资源保存的副本文件

### 缓存的优点

- 减少重复数据请求，避免通过网络再次加载资源，节省流量
- 降低服务器压力，提升网站性能
- 加快客户端加载网页的速度，提升用户体验

### 强缓存 和 协商缓存的区别

#### 强缓存

- 不需要给服务器发请求
- 没有发送请求 但是状态码是200
- 资源缓存有效时间 Expires（http1.0）Cache-Control(http1.1)
  - Cache-Control
    - max-age 有效时间
    - no-cache 不使用本地缓存，需要使用协商缓存
    - no-store 直接禁止浏览器缓存数据。每次请求资源都会向服务器要完整的资源
    - public 可以被所有用户缓存，包括终端用户和cdn等中间件代理服务器
    - private 只能被终端用户的浏览器缓存

#### 协商缓存

- 最终由服务器决定是否缓存
- 仅发送一次请求 状态码是304
- Last-Modified 第一次 If-Modified-Since 后续
  - 判断文件最后修改时间 没变 返回304 浏览器直接使用本地缓存 变了返回资源内容以及新的Last-Modified 更新本地缓存 下次请求使用新的Last-Modified
  - 缺点
    - 时间精度问题：1s以内修改内容 时间没变误判使用缓存
    - 内容准度问题：打开文件没有修改内容 或者修改了又该回去 重新保存 时间变了 内容没变 误判不适用缓存
- Etag 第一次 If-None-Match 后续
  - 服务器生成hash标识串 资源变化值就变化 当资源没有变化的时候 服务器依旧会将Etag返回（旧值）
  - http中没有指定如何生成etag 由开发者自行生成 hash 是比较理想的选择
  - 缺点
    - etag需要计算hash 如果文件过大计算会很慢 
    - 服务器集群 不同服务器计算hash值的方式可能不同  导致相同资源没命中缓存

### Service Worker

- 浏览器和服务端之间的代理服务器
- 主要功能是实现离线缓存
- 实现原理 拦截浏览器请求并返回缓存的资源文件
- 实现流程
  - 使用register注册serviceWorker脚本，浏览器获取到脚本后会解析然后进行安装
  - 通过监听install事件来监听安装，当安装完成后激活脚本
  - 激活脚本后 可以监听fetch事件来拦截请求并加载缓存的资源
- 限制
  - 无法访问DOM，可以通过postMessage 接口发送的消息来和控制页面进行通信
  - 只能在本地环境下 或 https网站中使用
  - 作用域限制 一个serviceWorker脚本只能作用于当前路径及其子路径
  - 兼容性不好

## dom树怎么生成的

- 浏览器是多进程框架，分别有主进程，第三方插件进程，GPU进程，渲染进程
  - Browser进程 
    - 负责、界面显示、前进、后退、页面管理、渲染bitmap位图 绘制到用户界面上
    - 网络资源管理、下载
  - 第三方插件进程 使用每种类型的插件时创建一个对应进程
  - GPU进程 至多一个 用于3d绘制
  - 渲染进程 通常说的浏览器内核
    - renderer进程 内部是多线程的 页面渲染 js脚本执行 事件处理

## 解析器遇到script标签

- 解析器约到script标签会停止dom树加载 执行script中的脚本或者下载js文件
- script标签 含有async 和defer 会异步加载不会阻塞dom树生成

## 渲染进程包括哪些进程

- GUI渲染线程
  - 渲染界面，解析html，css，构建dom树和renderObject树，布局layout，绘制paint
  - 重绘 回流 时 线程执行
  - gui渲染线程 和 js引擎线程是互斥的 js引擎执行时gui会被挂起 gui的更新会被保存在一个队列中等待js引擎空闲时立即执行
- js引擎线程 单线程
  - js内核 例如chrome v8
  - 解析js脚本运行代码
  - js引擎一直等待任务队列中任务的到来再进行处理，每个tab页中始终只有一个js线程再运行js程序
  - 与gui线程互斥 如果js引擎执行时间过长会造成页面卡顿
- 事件触发线程
  - 归属于渲染进程 而不是js引擎 控制`事件轮询` 可以理解 js 引擎忙不过来的时候需要浏览器另开线程协助
  - 鼠标点解 ajax异步请求 会将对应任务添加到事件触发线程中
  - 当对应的事件符合触发条件被触发时，该线程会把事件添加到待处理任务队列的队尾等待js引擎执行
  - js是单线程 所有处理队列的事件都得排队等待js引擎处理
- 定时触发器线程
  - 定时器setInterval 和 setTimeout所在线程
  - 浏览器定时计数器不是由js引擎计数的 如果任务队列阻塞线程状态就会影响计时的准确性
  - 计时完毕后添加到事件队列中等待js引擎空闲后执行
  - setTimrout低于4ms 按照4ms执行

- 异步http请求线程
  - 用于处理请求XMLHttpRequest ，在连接后是通过浏览器新开一个线程请求，如ajax，是浏览器新开的一个http线程
  - ajax状态变更时，将设置的回调函数加到事件队列由js引擎执行

## GUI渲染线程与js引擎互斥的原因

- 因为js可以操纵DOM，如果再修改这些元素属性同时渲染界面 （即js线程和GUI线程同时运行），那么渲染线程前后获得的元素数据就可能不一致了

## js引擎单线程的原因

- 当一个线程某个dom节点上增加内容 ，另一个线程将此dom节点删除这时就会造成一个比较尴尬的情况。
- web worker 允许js脚本创建多个线程 但是子线程完全受主线程控制，且不得操作dom，所以web worker不影响js是单线程的本质

## js执行顺序同步异步的产因

- 如果cpu计算量过大忙不过来 那只能等待 但是如果IO设备很慢比如ajax操作网络读取数据，不得不等结果出来在执行，那么可以直接将IO设备的任务挂起，先运行排在后面的任务等IO设备返回了结果再去执行之前挂起的任务
- 同步任务 在主线程排队执行的任务 只有钱一个任务执行完毕才能执行后一个任务
- 异步任务 不进入主线程 而进入任务队列 只有任务队列通知主线程某个异步任务可以执行了 该任务才会进入主线程被执行

## 为什么要单独的定时器线程？

- 因为JavaScript引擎是单线程的, 如果处于阻塞线程状态就会影响记计时的准确，因此很有必要单独开一个线程用来计时。

## http

- 应用层协议
- HTTP 是一个传输协议
- HTTP 常被用于在 Web 浏览器和网站服务器之间传递信息，以明文方式发送内容，不提供任何方式的数据加密

## https

- HTTP 传递信息是以明文的形式发送内容，这并不安全。而 HTTPS 出现正是为了解决 HTTP 不安全的特性
- HTTPS = HTTP + SSL/TLS
  - SSL 协议位于 TCP/IP 协议与各种应用层协议之间，浏览器和服务器在使用 SSL 建立连接时需要选择一组恰当的加密算法来实现安全通信，为数据通讯提供安全支持

## UDP

- 用户数据包协议，是一个简单的「面向数据报的通信协议」，即对应用层交下来的报文，不合并，不拆分，只是在其上面加上首部后就交给了下面的网络层

## TCP

- 传输控制协议，是一种可靠、「面向字节流的通信协议」，把上面应用层交下来的数据看成无结构的字节流来发送
  `TCP 应用场景适用于对效率要求低，对准确性要求高或者要求有链接的场景，而UDP 适用场景为对效率要求高，对准确性要求低的场景`
  `TCP适用于 强调准确性不强调效率`
  `UDP适用于 强调效率 对准确性要求低`

## DNS

- 应用层协议
- 主要负责域名解析

#### 在地址栏里输入一个 URL,到这个页面呈现出来，中间会发生什么？

- 找 ip 缓存->hosts 文件->dns->得到 ip+端口—>http 请求->tcp 包->服务器解析得到 html 树->构建 dom 树和 cssom 树->构建途中遇到 js 和 css->合并两个树然后进行页面布局
- 1 DNS 域名解析（此处涉及 DNS 的寻址过程），找到网页的存放服务器；

- 2 浏览器与服务器建立 TCP 连接；

- 3 浏览器发起 HTTP 请求；

- 4 服务器响应 HTTP 请求，返回该页面的 HTML 内容；

- 5 浏览器解析 HTML 代码，并请求 HTML 代码中的资源（如 JavaScript、CSS、图片等，此处可能涉及 HTTP 缓存）；

- 6 得到一颗 dom 树 解析 css 代码 得到带 css 的 dom 树
- 7 通过 css 属性计算每个元素的位置 方便进行排版

- 8 排版完成之后进行绘制 浏览器对页面进行渲染呈现给用户（此处涉及浏览器的渲染原理）。

## 有限状态机 处理字符串

- 每一个状态都是一个机器
  - 在每一个机器里我们可以做 计算 存储 输出
  - 所有的这些机器接受的输入是一致的
  - 状态机的诶一个机器本身没有状态，如果我们用函数来表示的话，它应该是纯函数
- 每一个机器知道下一个状态
  - 每个机器都有确定的下一个状态 moore
  - 每个机器根据输入决定下一个状态 mealy

## mearly 型 有限状态机

- 根据输入值返回不同的状态 这就意味着需要返回一个函数

```js
function match(string) {
  let state = start;
  for (let s of string) {
    state = state(s);
  }
  return state == end;
}
function start(s) {
  if (s == 'a') {
    return foundB;
  }
  return start(s);
}
function end() {
  return end;
}
function foundB(s) {
  if (s == 'b') {
    return foundC;
  }
  return start(s);
}
function foundC(s) {
  if (s == 'c') {
    return foundD;
  }
  return start(s);
}
function foundD(s) {
  if (s == 'd') {
    return foundE;
  }
  return start(s);
}
function foundE(s) {
  if (s == 'e') {
    return foundF;
  }
  return start(s);
}
function foundF(s) {
  if (s == 'f') {
    return end;
  }
  return start(s);
}

console.log(match('dsabcdefsdasdsa'));
```
